3 ОПИС ВИКОНАНОЇ РОБОТИ

Мета використання: отримання теоретичних відомостей про патерн проєктування «Адаптер» (Adapter) з каталогу GoF, уточнення його структури, ролей учасників, принципів роботи та варіантів реалізації; формування прикладу програмної реалізації на C#; проведення порівняльного аналізу з іншими патернами проєктування.

3.1 Загальна характеристика патернів GoF

Патерни GoF — це набір узагальнених рішень типових задач у об'єктно-орієнтованому програмуванні, описаний у книзі «Design Patterns» чотирма авторами (Gang of Four). Вони не є готовим кодом, а виступають як шаблони проєктування, які можна адаптувати під конкретну задачу.
Усі патерни поділяються на три групи: породжувальні (відповідають за створення об'єктів), структурні (визначають організацію класів і зв’язків між ними) та поведінкові (описують взаємодію об’єктів).
Використання цих патернів допомагає створювати більш гнучкі, зрозумілі та підтримувані програмні системи, а також спрощує комунікацію між розробниками завдяки спільній термінології.

3.2 Визначення патерну «Адаптер»

Патерн «Адаптер» (Adapter) є структурним патерном проєктування з каталогу GoF. Згідно з класичним визначенням, «Адаптер перетворює інтерфейс одного класу на інший інтерфейс, очікуваний клієнтом. Адаптер дозволяє класам із несумісними інтерфейсами працювати разом».
Аналогію можна провести з реального життя: електричний адаптер (перехідник) дозволяє підключити пристрій з одним типом вилки до розетки іншого стандарту. Сам пристрій при цьому не змінюється — змінюється лише спосіб взаємодії з ним.
У програмуванні адаптер виконує подібну роль: він «обгортає» існуючий клас і надає новий інтерфейс, який відповідає очікуванням клієнта. Завдяки цьому можна повторно використовувати вже існуючий код без його модифікації.

3.3 Проблема, яку вирішує патерн «Адаптер»

Уявіть, що ви робите програму для торгівлі на біржі. Ваша програма завантажує біржові котирування з декількох джерел в XML, а потім малює красиві графіки.
У якийсь момент ви вирішуєте покращити програму, застосувавши сторонню бібліотеку аналітики. Але біда - бібліотека підтримує тільки формат даних JSON, несумісний з вашим додатком.
Ви могли б переписати бібліотеку, щоб вона підтримувала формат XML. Але, по-перше, це може порушити роботу існуючого коду, який залежить від бібліотеки. А по-друге, у вас може просто не бути доступу до її вихідного коду.
Ви можете створити адаптер. Це об'єкт-перекладач, який трансформує інтерфейс або дані одного об'єкта на такий вигляд, щоб він став зрозумілим іншому об'єкту.
При цьому адаптер обертає один з об'єктів, тому інший об'єкт навіть не знає про наявність першого. Наприклад, ви можете обернути об'єкт, що працює в метрах, адаптером, який конвертував би дані в фути.
Адаптери можуть не тільки переводити дані з одного формату до іншого, але й допомагати об'єктам з різними інтерфейсами працювати спільно. Це працює так:
Адаптер має інтерфейс, який сумісний з одним із об'єктів.
Тому цей об'єкт може викликати методи адаптера.
Адаптер отримує ці виклики і перенаправляє їх другого об'єкта, але вже в тому форматі та послідовності, які зрозумілі другому об'єкту.
Іноді можна створити навіть двосторонній адаптер, який працював би в обидві сторони.
Таким чином, у додатку біржових котирувань ви могли б створити клас XML_To_JSON_Adapter, який би обертав об'єкт того чи іншого класу бібліотеки аналітики. Ваш код надсилав би адаптеру запити у форматі XML, а адаптер спочатку транслював вхідні дані у формат JSON, а потім передавав їх методам оберненого об'єкта аналітики.

3.4 Структура патерну «Адаптер»

Ця реалізація використовує агрегацію: об'єкт адаптера "обертає", тобто містить посилання на службовий об'єкт. Такий підхід працює у всіх мовах програмування.
1.	Клієнт – це клас, який містить існуючу бізнес-логіку програми.
2.	Клієнтський інтерфейс визначає протокол, через який клієнт може працювати з іншими класами.
3.	Сервіс це якийсь корисний клас, зазвичай сторонній. Клієнт неспроможна використовувати цей клас безпосередньо, оскільки сервіс має незрозумілий йому інтерфейс.
4.	Адаптер - це клас, який може одночасно працювати і з клієнтом, і сервісом. Він реалізує клієнтський інтерфейс і містить посилання об'єкт сервісу. Адаптер отримує виклики від клієнта через методи клієнтського інтерфейсу, а потім переводить їх у виклики методів оберненого об'єкта у правильному форматі.
5.	Працюючи з адаптером через інтерфейс, клієнт не прив'язується до конкретного класу адаптера. Завдяки цьому ви можете додавати до програми нові види адаптерів, незалежно від клієнтського коду. Це може стати в нагоді, якщо інтерфейс сервісу раптом зміниться, наприклад, після виходу нової версії сторонньої бібліотеки.

3.5 Ситуації застосування патерну «Адаптер»

Патерн «Адаптер» доцільно використовувати у таких випадках:
По-перше, коли потрібно інтегрувати сторонню бібліотеку або застарілий код, який має несумісний інтерфейс.
По-друге, коли необхідно повторно використати існуючий клас, але його інтерфейс не відповідає новим вимогам.
По-третє, при роботі з різними системами або сервісами, які мають різні формати даних або API
По-четверте, при переході на нову архітектуру, коли потрібно забезпечити сумісність зі старими компонентами.
Приклади з реального світу:
•	адаптери для роботи з різними базами даних;
•	обгортки над сторонніми API;
•	конвертери форматів (наприклад, XML → JSON);
•	інтеграція платіжних систем.

3.6 Переваги та недоліки патерну «Адаптер»

Переваги:
•	забезпечує повторне використання існуючого коду без його зміни;
•	зменшує зв'язність між клієнтом і конкретними реалізаціями;
•	підвищує гнучкість та розширюваність системи;
•	дозволяє легко інтегрувати сторонні компоненти.
Недоліки:
•	збільшує кількість класів у системі;
•	може ускладнювати розуміння коду через додатковий рівень абстракції;
•	при великій кількості адаптерів система може стати перевантаженою.

3.7 Приклад реалізації патерну (узагальнений)

Припустимо, клієнт очікує інтерфейс ITarget з методом Request(), але існує клас Adaptee, який має метод SpecificRequest().
Адаптер реалізує ITarget і всередині викликає SpecificRequest(), перетворюючи виклик у потрібний формат.
У результаті клієнт працює з ITarget, не знаючи про існування Adaptee, а адаптер забезпечує сумісність між ними.

3.8 Порівняння з іншими патернами

Патерн «Адаптер» часто порівнюють з Facade та Decorator.
Adapter змінює інтерфейс об'єкта, щоб зробити його сумісним з очікуваннями клієнта.
Facade спрощує інтерфейс складної підсистеми.
Decorator розширює функціональність об'єкта без зміни його інтерфейсу.
Таким чином:
•	Adapter — перетворює інтерфейс,
•	Facade — спрощує інтерфейс,
•	Decorator — розширює поведінку.

Під час роботи було з’ясовано, що патерн «Адаптер» призначений для забезпечення взаємодії між класами з несумісними інтерфейсами шляхом створення проміжного об’єкта, який виконує перетворення викликів. Його використання дозволяє інтегрувати існуючі компоненти або сторонні бібліотеки без внесення змін до їхнього коду.
Було проаналізовано структуру патерну.
Окрему увагу приділено задачам, які вирішує цей патерн, зокрема зменшенню жорсткої зв’язаності між компонентами системи та підвищенню її гнучкості. Наведено типові сценарії використання, такі як інтеграція сторонніх API, робота з різними форматами даних та підтримка сумісності зі старими системами.
Крім того, було визначено основні переваги патерну, серед яких повторне використання коду та розширюваність системи, а також недоліки, пов’язані зі збільшенням кількості класів і ускладненням структури.
Отримані результати дозволяють зробити висновок, що патерн «Адаптер» є важливим інструментом у розробці програмного забезпечення, особливо в умовах інтеграції різнорідних компонентів, і може ефективно застосовуватися при побудові сучасних програмних систем.

1 // Цільовий інтерфейс (Target)
2 interface ITarget
3 {
4     string Request();
5 }
6 // Існуючий клас з несумісним інтерфейсом (Adaptee)
7 class Adaptee
8 {
9     public string SpecificRequest()
10     {
11         return "Специфічний запит";
12     }
13 }
14 // Клас Адаптер
15 class Adapter : ITarget
16 {
17     private Adaptee adaptee;
18
19     public Adapter(Adaptee adaptee)
20     {
21         this.adaptee = adaptee;
22     }
23
24     public string Request()
25     {
26         // Перетворення виклику
27         return "Адаптований: " + adaptee.SpecificRequest();
28     }
29 }
30 // Клієнт
31 class Program
32 {
33     static void Main()
34     {
35         Adaptee adaptee = new Adaptee();
36         ITarget target = new Adapter(adaptee);
37
38         Console.WriteLine(target.Request());
39     }
40 }
